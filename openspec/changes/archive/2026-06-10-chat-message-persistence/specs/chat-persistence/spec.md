# Chat Message Persistence Specification

## Purpose

Enable full persistence of chat messages in SQLite while preserving the complete AI SDK `UIMessage` structure (including tool call metadata) across page reloads, so users can resume conversations and the backend has complete context for multi-agent orchestration.

## Requirements

### Requirement: Messages Table Schema

The system MUST store chat messages in a SQLite table with the following structure:

| Column | Type | Constraint | Purpose |
|--------|------|-----------|---------|
| `id` | INTEGER | PRIMARY KEY AUTOINCREMENT | Unique message identifier |
| `chat_id` | INTEGER | NOT NULL, FOREIGN KEY → `chats(id)` ON DELETE CASCADE | Links message to chat |
| `role` | TEXT | NOT NULL, CHECK(role IN ('user', 'assistant', 'system', 'tool')) | Message origin/type |
| `message_data` | TEXT | NOT NULL | Complete `UIMessage` serialized as JSON |
| `created_at` | INTEGER | NOT NULL, DEFAULT (unixepoch()) | Unix timestamp of creation |

The system MUST drop deprecated columns `toolName` and `toolResult` if they exist (schema migration).

#### Scenario: Schema replacement preserves data integrity

- GIVEN the old schema with columns `toolName`, `toolResult`
- WHEN the migration runs
- THEN:
  - Old data is NOT lost or corrupted
  - New `message_data` column exists and is NOT NULL for all existing rows
  - Deprecated columns are removed
  - Foreign key constraint on `chat_id` enforces referential integrity

### Requirement: Message Serialization

The system MUST store the complete `UIMessage` object (from Vercel AI SDK v6) as a JSON string in the `message_data` column without modification or loss of fields.

The serialized message MUST preserve:
- `id`: string unique identifier generated by AI SDK or client
- `role`: one of 'user' | 'assistant' | 'system' | 'tool'
- `parts`: array of message parts, each containing:
  - For text parts: `{ type: 'text', text: string }`
  - For tool-use parts: `{ type: string (format 'tool-{toolName}'), toolCallId: string, state: string, input: object, output?: object }`
  - All other fields present in the original `UIMessage`

#### Scenario: UIMessage with tool parts is round-tripped without data loss

- GIVEN a `UIMessage` with a tool-use part containing `toolCallId: 'call_abc123'`, `input: { domain: 'example.com' }`, `output: { score: 89, issues: [...]}`
- WHEN the message is serialized to JSON and stored in `message_data`
- AND THEN deserialized by parsing `message_data` back to an object
- THEN:
  - `toolCallId` equals 'call_abc123'
  - `input.domain` equals 'example.com'
  - `output.score` equals 89
  - All properties are present (no truncation or filtering)

### Requirement: Save Message to Database (UPSERT)

The system MUST provide a function to insert or update a message in the database, keyed by `message.id` (deduplication).

The function signature MUST be:
```typescript
saveMessage(chatId: number, message: UIMessage): Promise<{ success: boolean; error?: string }>
```

If a message with the same `id` already exists in the same `chat_id`, the row MUST be updated (UPSERT), not duplicated.

#### Scenario: Retry of onFinish callback does not create duplicate messages

- GIVEN a message with `id: 'msg_abc123'` is saved successfully
- AND the message is queried from the database
- WHEN `saveMessage(chatId, sameMessage)` is called again (e.g., due to network retry)
- THEN:
  - Only one row with `id: 'msg_abc123'` exists in the database for that `chat_id`
  - The row count is 1, not 2

### Requirement: Load Messages by Chat ID

The system MUST provide a function to retrieve all messages for a given chat, ordered by creation time.

The function signature MUST be:
```typescript
getMessagesByChat(chatId: number): Promise<UIMessage[]>
```

The function MUST:
- Return messages ordered by `created_at` ascending (oldest first)
- Deserialize `message_data` JSON string back to `UIMessage` objects
- Return an empty array if the chat has no messages
- Throw an error if the `chatId` does not exist

#### Scenario: Chat history is loaded and deserialized in order

- GIVEN a chat with three messages created at timestamps T1, T2, T3 (T1 < T2 < T3)
- WHEN `getMessagesByChat(chatId)` is called
- THEN:
  - An array of three `UIMessage` objects is returned
  - The order is [message_T1, message_T2, message_T3]
  - Each message is a valid `UIMessage` (role, parts, id are present)

### Requirement: GET /api/chats/[id] Endpoint

The system MUST respond to a GET request to `/api/chats/[id]` by returning the chat details and its complete message history.

**Response Shape (200 OK):**
```json
{
  "chat": {
    "id": number,
    "projectId": number,
    "title": string,
    "createdAt": string (ISO 8601)
  },
  "messages": [
    {
      "id": string,
      "role": "user" | "assistant" | "system" | "tool",
      "parts": [...]
    }
  ]
}
```

**Response (404 Not Found):**
```json
{
  "error": "Chat not found"
}
```

The endpoint MUST:
- Verify that the requested `chatId` exists
- Return the chat metadata and all associated messages
- Deserialize message_data from the database into complete `UIMessage` objects
- Order messages by creation time (oldest first)

#### Scenario: Chat with complete history is retrieved

- GIVEN a chat with id=1, title="Estrategia SEO Q2", and two user-assistant message pairs
- WHEN a GET request is made to `/api/chats/1`
- THEN:
  - Status is 200
  - Response includes `chat.id = 1`
  - Response includes `chat.title = "Estrategia SEO Q2"`
  - Response includes `messages` array with 4 `UIMessage` objects in order
  - All tool-call metadata in any tool parts is intact

#### Scenario: Nonexistent chat returns 404

- GIVEN a chat id=999 does not exist
- WHEN a GET request is made to `/api/chats/999`
- THEN:
  - Status is 404
  - Response is `{ "error": "Chat not found" }`

### Requirement: POST /api/chats/sendMessage Endpoint

The system MUST respond to a POST request to `/api/chats/sendMessage` by persisting the message to the database and returning a success or error response.

**Request Body:**
```json
{
  "chatId": number,
  "message": UIMessage
}
```

**Response (200 OK):**
```json
{
  "success": true
}
```

**Response (400 Bad Request) - Missing Required Fields:**
```json
{
  "success": false,
  "error": "chatId and message are required"
}
```

**Response (404 Not Found) - Chat Does Not Exist:**
```json
{
  "success": false,
  "error": "Chat not found"
}
```

**Response (500 Internal Server Error) - Database Error:**
```json
{
  "success": false,
  "error": "Failed to save message"
}
```

The endpoint MUST:
- Validate that `chatId` is a positive integer and `message` is a valid `UIMessage`
- Call `saveMessage(chatId, message)` to persist the message
- Return 200 with `{ success: true }` on successful insert/update
- Return 404 if the chat does not exist
- Return 400 if validation fails
- Return 500 if the database operation fails unexpectedly

#### Scenario: User message is persisted successfully

- GIVEN a chat with id=1 exists
- AND a `UIMessage` with role='user' and a text part is prepared
- WHEN a POST request is made to `/api/chats/sendMessage` with `{ chatId: 1, message }`
- THEN:
  - Status is 200
  - Response is `{ "success": true }`
  - The message is queryable from `getMessagesByChat(1)`
  - The message_data in the database matches the serialized input

#### Scenario: Invalid request returns 400

- GIVEN a POST request to `/api/chats/sendMessage` with missing `chatId`
- WHEN the endpoint processes the request
- THEN:
  - Status is 400
  - Response is `{ "success": false, "error": "chatId and message are required" }`

#### Scenario: Nonexistent chat returns 404

- GIVEN a POST request to `/api/chats/sendMessage` with `chatId: 999` (does not exist)
- WHEN the endpoint processes the request
- THEN:
  - Status is 404
  - Response is `{ "success": false, "error": "Chat not found" }`

### Requirement: ChatArea Component Accepts chatId Prop

The `ChatArea` component MUST accept a `chatId` prop (number) and use it to load and persist messages.

**Props:**
```typescript
interface ChatAreaProps {
  chatId: number;
  // existing props...
}
```

The component MUST:
- Accept and store the `chatId` prop
- Load message history on mount via `GET /api/chats/[chatId]`
- Pass loaded messages as `initialMessages` to the `useChat` hook
- Pass the `chatId` to the `useChat` hook's body parameter so the AI API can identify the chat session

#### Scenario: ChatArea loads initial messages on mount

- GIVEN `ChatArea` is mounted with `chatId=1`
- AND chat 1 has 2 existing messages in the database
- WHEN the component mounts and fetches from `GET /api/chats/1`
- THEN:
  - The `useChat` hook is initialized with `initialMessages` containing 2 `UIMessage` objects
  - The chat history is displayed in the UI

### Requirement: ChatArea Persists Messages on Stream Finish

The `ChatArea` component MUST configure the `useChat` hook's `onFinish` callback to persist each message after the stream completes.

The component MUST:
- Implement an `onFinish` callback that calls `POST /api/chats/sendMessage` with the completed `UIMessage`
- Pass the `chatId` and message to the endpoint
- Handle errors gracefully (e.g., log to console; user sees the message in UI even if save fails for MVP)
- Persist only when the stream finishes (not during streaming)

#### Scenario: Assistant message is persisted after stream finishes

- GIVEN the user sends a message in `ChatArea` with `chatId=1`
- AND the AI SDK finishes streaming the assistant response (onFinish fires)
- WHEN the `onFinish` callback executes
- THEN:
  - A POST request is made to `/api/chats/sendMessage` with the complete assistant `UIMessage`
  - The message is saved in the database
  - The user sees the message in the chat UI

#### Scenario: User message is already persisted by the client or AI SDK

- GIVEN the user types a message and clicks send
- WHEN the user message is sent via `useChat`
- THEN:
  - The user message appears in the local `useChat` state immediately
  - The client MAY call `POST /api/chats/sendMessage` for the user message (optional persistence)
  - The user experience is not blocked by database latency

### Requirement: ChatLayout Passes chatId to ChatArea

The `ChatLayout` component MUST pass the `chatId` to `ChatArea` so that `ChatArea` can load and persist messages.

**Props Flow:**
```
ChatLayout receives chatId → passes chatId to ChatArea → ChatArea uses chatId
```

#### Scenario: chatId is threaded from route to component

- GIVEN `ChatLayout` is rendered with access to the chat id (from URL params or state)
- WHEN `ChatLayout` renders the `ChatArea` child component
- THEN:
  - `chatId` is passed as a prop to `ChatArea`
  - `ChatArea` can load and save messages for this specific chat

### Requirement: Message Lifecycle Atomicity

The system MUST ensure that each message save operation is atomic at the database level (all-or-nothing).

#### Scenario: Message save is atomic

- GIVEN a message is being saved to the database
- WHEN the database connection is interrupted mid-insert
- THEN:
  - Either the entire message is persisted (with all columns populated)
  - OR no row is created (rollback/constraint violation)
  - There MUST NOT be a partial row with NULL message_data or missing fields

### Requirement: Message Deserialization Validation

The system MUST validate that deserialized messages from `message_data` conform to the `UIMessage` type before returning them to the frontend.

#### Scenario: Corrupted message_data is handled gracefully

- GIVEN a row exists with invalid JSON in `message_data` (e.g., truncated JSON string)
- WHEN `getMessagesByChat(chatId)` attempts to deserialize this row
- THEN:
  - Either the row is skipped with a warning logged
  - OR an error is thrown and propagated to the caller
  - The request MUST NOT crash the server

## Non-Goals

The following are explicitly out of scope for this change:

- **Pagination or lazy-loading**: All messages are loaded at once (MVP pragmatism).
- **Message editing or deletion**: Users cannot modify or remove messages.
- **System message persistence**: System messages are not used in this MVP.
- **Incremental streaming to database**: Messages are saved only on stream completion.
- **Schema versioning**: No versioning mechanism; data migration will be manual if AI SDK v7 changes `UIMessage` structure.
- **Automatic data migration**: Existing messages (if any) are not automatically migrated.
- **Tool call query indices**: SQLite JSON functions can be used later if querying tool calls by metadata is needed.

## Rules

- All database operations MUST use the Drizzle ORM (not raw SQL queries).
- All endpoint responses MUST return proper HTTP status codes (200, 400, 404, 500).
- All TypeScript code MUST pass `pnpm exec tsc --noEmit` without errors.
- All code MUST pass `pnpm exec @biomejs/biome check --apply .` without errors.
- Every requirement MUST have at least one testable Given/When/Then scenario.
- RFC 2119 keywords (MUST, SHOULD, MAY) MUST be used consistently to denote requirement strength.
